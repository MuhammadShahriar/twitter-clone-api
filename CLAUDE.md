# CLAUDE.md

Guidance for Claude Code (and humans) working in this repository.

## What this is

`twitter-clone-api` — an ASP.NET Core (.NET 10) Web API for a Twitter clone, built on **Clean Architecture** with **CQRS via MediatR**. This is **Module 0: the walking skeleton** — one vertical slice (create + list a tweet) proving the spine end-to-end and deploying to Render. Keep it minimal and production-shaped; do not add features beyond what a module specifies.

## Solution layout

Four projects under `src/`, wired with the Clean Architecture dependency rule (dependencies point inward only):

- **TwitterClone.Domain** — entities (`Tweet`, `BaseEntity`). No external dependencies. The innermost layer.
- **TwitterClone.Application** — use cases as MediatR requests, organised by feature under `Tweets/`. Holds the data-access abstractions (`IRepository<T>`, `ITweetRepository`, `IUnitOfWork`), FluentValidation validators, and the `ValidationBehaviour` pipeline. Depends only on Domain — **EF Core is invisible here** (no `DbContext`, no `DbSet`, no `IQueryable`).
- **TwitterClone.Infrastructure** — `ApplicationDbContext` (EF Core + Npgsql, concrete and confined to this layer), the repository/unit-of-work implementations (`Repository<T>`, `TweetRepository`, `UnitOfWork`), entity configurations, migrations, `ConnectionStringResolver`, and **ASP.NET Core Identity** (the `ApplicationUser` entity under `Identity/`). Depends on Application.
- **TwitterClone.Api** — controllers, DI composition (`Program.cs`), Swagger, exception handling. Depends on Application + Infrastructure. The only runnable project.
- **tests/TwitterClone.IntegrationTests** — xUnit tests. Two flavours: HTTP-level integration tests via `WebApplicationFactory<Program>` over the EF Core **in-memory** provider (the API spine, no live DB needed), and repository-level **SQLite (in-memory, relational)** tests via `SqliteTestHarness` for queries that must be validated against real SQL (see the testing policy below).

Solution file is `twitter-clone-api.slnx` (modern XML format; opens in Visual Studio 2026).

## Conventions

- **CQRS**: every use case is a MediatR `IRequest`. Commands mutate, queries read. One folder per use case: `Tweets/Commands/CreateTweet/` holds the command (record), handler, and validator together.
- **Controllers are thin**: they only build a request and call `ISender.Send(...)`. No business logic in the API layer.
- **Data access is Repository + Unit of Work**: handlers depend on `ITweetRepository` (entity-specific queries) / `IRepository<T>` and `IUnitOfWork` — **never** on `DbContext` or EF Core. Repositories **never** call `SaveChanges`; they only stage changes (`AddAsync`/`Update`/`Remove`). Only `IUnitOfWork.SaveChangesAsync` commits, so multiple repository operations land in one transaction. Repositories share the same scoped `DbContext` instance via DI. Reads use `AsNoTracking()` and push ordering/filtering down to the database; expose named methods (e.g. `GetAllNewestFirstAsync`) rather than leaking `IQueryable`.
- **Validation**: add a `FluentValidation.AbstractValidator<TCommand>` next to the command. The `ValidationBehaviour` runs it automatically; failures surface as RFC 7807 `400` responses via `ValidationExceptionHandler`.
- **DTOs**: return records (e.g. `TweetDto`) from handlers, never domain entities.
- **DI**: each layer exposes an `AddXxx()` extension (`AddApplication`, `AddInfrastructure`) called from `Program.cs`.
- **Auth / Identity** (Module 1): ASP.NET Core Identity is an **infrastructure/persistence concern** and must never leak into Domain or Application — those layers carry **no** reference to any Identity package. The `ApplicationUser` (extends `IdentityUser<Guid>`, adds the `Handle`/`DisplayName`/`Bio`/`CreatedAtUtc` profile fields) lives in `Infrastructure/Identity`; `ApplicationDbContext` derives from `IdentityDbContext<ApplicationUser, IdentityRole<Guid>, Guid>` (so `AspNetUsers.Id` is native `uuid`, matching `Tweet.Id`), and its `OnModelCreating` calls `base` **first**, then `ApplyConfigurationsFromAssembly`. The unique `Handle` index and profile column lengths are set in `ApplicationUserConfiguration`. Identity services are registered via `AddIdentityCore<ApplicationUser>().AddRoles<IdentityRole<Guid>>().AddEntityFrameworkStores<ApplicationDbContext>()` in `AddInfrastructure` (cookieless, API-friendly).
- **Auth use-cases & JWT** (Module 1B): the Application layer defines the abstractions `IIdentityService` (create user / validate credentials) and `IJwtTokenGenerator` (issue an `AccessToken`), speaking only in Identity-free models (`AuthUser`, `CreateUserResult`, `AccessToken`) under `Application/Common`. Infrastructure implements them: `Identity/IdentityService` (wraps `UserManager<ApplicationUser>`) and `Authentication/JwtTokenGenerator` (HMAC-SHA256, puts the user's Guid in the `sub` claim). Auth handlers depend **only** on those abstractions — never on `UserManager`/`ApplicationUser`. Register/login are CQRS commands under `Application/Authentication/Commands/...`; the thin `AuthController` exposes `POST /api/auth/register` (→ 200 `RegisterResult`) and `POST /api/auth/login` (→ 200 `AuthenticationResult` with the JWT, or **401** when `LoginCommandHandler` returns `null`). Identity failures (duplicate email/handle, weak password) are surfaced as a `ValidationException` → the existing 400 handler. JWT settings bind the `Jwt` config section into `JwtSettings`; `Jwt:Issuer`/`Audience`/`ExpiryMinutes` are committed in `appsettings.json` but **`Jwt:SecretKey` is a secret** — user-secret locally, `Jwt__SecretKey` env var on Render. **Fail-fast (Module 1 hardening):** `JwtSettings` is registered with `AddOptions<JwtSettings>().Bind(...).Validate(...).ValidateOnStart()` in `AddInfrastructure` — a missing/short signing key (must be ≥ 32 bytes / 256-bit) or a missing issuer/audience **aborts startup** with a clear message instead of a runtime 500 on first login. There is no plain `Configure<JwtSettings>` (the validated registration is the single binding point).
- **Login hardening: lockout + timing** (Module 1 hardening): login goes through `IIdentityService.ValidateCredentialsAsync`, which returns a `CredentialValidationResult` (`Success`/`InvalidCredentials`/`LockedOut`) — Identity-free, so the Application never sees `SignInResult`. `IdentityService` drives **account lockout** via `UserManager`'s lockout primitives (`IsLockedOutAsync`/`CheckPasswordAsync`/`AccessFailedAsync`/`ResetAccessFailedCountAsync`) — deliberately **not** `SignInManager`, to avoid pulling the ASP.NET Core web framework into the Infrastructure class library (consistent with the cookieless `AddIdentityCore`). Lockout policy (set in `AddInfrastructure`): **5 failed attempts → locked 15 min**, `AllowedForNewUsers = true` (so new accounts are lockout-enabled at creation). When the email is unknown, a **dummy password-hash verification** runs to equalize timing (user-enumeration defense). `LoginCommandHandler` throws `AccountLockedException` on `LockedOut` → `AccountLockedExceptionHandler` maps it to **423 Locked** (generic message); invalid credentials still return a generic **401** (no user-vs-password distinction). **Rate limiting on the auth endpoints was intentionally deferred** as a defense-in-depth follow-up (lockout is the primary control).
- **Register race backstop** (Module 1 hardening): `IdentityService.CreateUserAsync` keeps the fast `AnyAsync` handle pre-check, and additionally **catches the Npgsql unique-violation (SQLSTATE 23505)** from `UserManager.CreateAsync` and returns a `CreateUserResult.Failure` — so a concurrent/colliding handle registration that slips past the pre-check surfaces as a clean **400**, never an unhandled 500.
- **JWT validation & current user** (Module 1C-i): `Api/Common/JwtAuthenticationExtensions.AddJwtBearerAuthentication()` registers JWT bearer auth whose `TokenValidationParameters` are bound from `IOptions<JwtSettings>` (resolved lazily) so they can't drift from what `JwtTokenGenerator` signs. `MapInboundClaims = false` keeps JWT claim names intact (`sub`, `handle`). Pipeline order in `Program.cs` is **CORS → `UseAuthentication` → `UseAuthorization` → `MapControllers`** (CORS must run first). The Application defines `ICurrentUserService` (`UserId`/`Handle`/`IsAuthenticated`, all HTTP/JWT-free); its implementation `Api/Common/CurrentUserService` reads the validated principal via `IHttpContextAccessor` and lives in the **API** layer. Protect an endpoint with `[Authorize]`; `GET /api/auth/me` is the reference protected route (→ 401 without a token), backed by the `GetCurrentUser` query which resolves the caller via `ICurrentUserService` then re-reads the profile via `IIdentityService`.
- **Refresh tokens & cookies** (Module 1C-ii): strategy is **access token in client memory (15 min) + refresh token in an httpOnly cookie (7 days)**. The `RefreshToken` domain entity (in `Domain/Entities`, references its owner by `UserId` value only — the FK to `AspNetUsers` is configured in Infrastructure) is persisted **hashed** (SHA-256 hex); the raw token lives only in the cookie and is never stored or logged. Login starts a `FamilyId` lineage; `POST /api/auth/refresh` **rotates** (revoke current, issue successor in the same family); replaying a revoked token is **reuse detection** → the whole family is revoked and the call 401s; `POST /api/auth/logout` revokes the family + clears the cookie. Refresh/logout do **not** require an access token (it may be expired) — they rely on the cookie. New abstractions: `IRefreshTokenService` (generate/hash, Infrastructure crypto) and `IRefreshTokenRepository` (tracked lookups by hash/family); handlers return `AuthTokens` (body DTO + raw refresh token), and the **controller** owns all cookie read/write via `AuthCookieSettings`. Config: access-token lifetime is `Jwt:ExpiryMinutes` (15), refresh lifetime `RefreshToken:ExpiryDays` (7); cookie `AuthCookie` (`Name`/`Path`/`SameSite`/`Secure`/`Domain`) is **environment-driven** — prod `SameSite=None;Secure=true` (cross-site SPA), dev `SameSite=Lax;Secure=false` (in `appsettings.Development.json`). CORS adds **`AllowCredentials()`** with the explicit origin list (incompatible with `AllowAnyOrigin`) so the browser will send the cookie.
- **Tweets owned by the author** (Module 1D): a `Tweet` carries `Guid AuthorId` (the owner's `AspNetUsers.Id`) and **no** `AuthorHandle` — its constructor requires `(content, authorId)`. The Domain stays Identity-free: there is **no** navigation from `Tweet` to `ApplicationUser`; the FK to `AspNetUsers` (+ an index on `AuthorId`, `OnDelete: Cascade`) is configured in `TweetConfiguration` in Infrastructure only. `POST /api/tweets` is **`[Authorize]`-protected** (→ **401** without a token) and `CreateTweetCommand` takes only `content` — `CreateTweetCommandHandler` reads the author from `ICurrentUserService.UserId` (never the request body; defensive 401 guard) then re-reads via the author join for the response. `TweetDto` now carries `AuthorId`/`AuthorHandle`/`AuthorDisplayName`; the read side gets these from a **join to `AspNetUsers` done in `TweetRepository`** (Infrastructure), which projects straight into `TweetDto` (`AsNoTracking`, DB-side ordering) — so `ITweetRepository`'s read methods (`GetAllNewestFirstAsync`, `GetByIdWithAuthorAsync`) return DTOs, not entities, and Identity never crosses into Application. **`GET /api/tweets` and `/{id}` stay public.** Migration `LinkTweetToAuthor` **deletes pre-existing tweet rows** (created before auth, no honest author to backfill) before adding the non-nullable `AuthorId` — pre-release demo data, so no manual DB action is needed beyond running the migration.
- **Replies, delete & cursor pagination** (Module 2A): a `Tweet` carries a nullable `Guid? ParentId` (self-reference — a reply points at the tweet it replies to; top-level tweets have `ParentId == null`); the constructor is `(content, authorId, parentId = null)`. The Domain stays Identity-free **and navigation-free**: no nav to the parent tweet either — the self-FK (+ index on `ParentId`, `OnDelete: Cascade`) is configured in `TweetConfiguration` only. **Reply counts are computed on the read side** (a correlated subquery in `TweetRepository.Project`, no denormalised counter, no N+1); `TweetDto` gained `ParentId` + `ReplyCount`. **Feed = top-level only** (`GetFeedAsync` filters `ParentId == null`, newest-first); **replies are read via `GET /api/tweets/{id}/replies`** (`GetRepliesAsync`, **oldest-first** for natural thread reading order) — both **cursor-paginated**: query params `cursor` (opaque) + `limit` (default 20, max 50, clamped by `PaginationDefaults`), response shape `CursorPage<TweetDto>` = `{ items, nextCursor }` (`nextCursor` null when no more). The cursor (`TweetCursor`, base64url `{ticks}:{id}`) is keyset paging on `(CreatedAtUtc, Id)` — `Id` is the stable tiebreaker so rows sharing a timestamp never duplicate/skip; over-fetch `limit + 1` to decide `nextCursor`. **Creating a reply**: `CreateTweetCommand` gains optional `ParentId`; the validator's `MustAsync` checks the parent exists (`ITweetRepository.ExistsAsync`) → missing parent is a **400**. **Delete**: `DeleteTweetCommand(id)` → `DELETE /api/tweets/{id}` (`[Authorize]`); **author-only** — the handler compares `tweet.AuthorId` to `ICurrentUserService.UserId` and throws `ForbiddenAccessException` (→ **403**, not 404) for someone else's tweet, `NotFoundException` (→ **404**) when missing, else **204**. Delete **cascades to direct replies**, removed **explicitly in the handler** (so it holds on the non-cascading in-memory test provider) with the DB self-FK cascade as a backstop for deeper chains. New Application exceptions `NotFoundException`/`ForbiddenAccessException` map via `NotFoundExceptionHandler`/`ForbiddenAccessExceptionHandler` (registered in `Program.cs` alongside the existing handlers). **`GET /api/tweets`, `/{id}`, and `/{id}/replies` stay public.** Migration `AddTweetReplies` (adds the `ParentId` column, its index, and the self-FK — additive, no data loss).
- **Image upload via Cloudinary** (Module 2D, backend): an authenticated user can attach **up to 4 images** to a tweet, **backend-proxied** to Cloudinary (the API holds the secret and does the upload). Cloudinary is an **Infrastructure** concern behind an **Application** abstraction `IImageStorageService` (`UploadAsync(ImageUpload) → ImageUploadResult`), speaking only provider-free models (`ImageUpload`/`ImageUploadResult` under `Application/Common/Models`) — the `CloudinaryDotNet` package (MIT) and `Cloudinary` client never leak past `Infrastructure/Media` (`CloudinaryImageStorageService`). **Media is a child of the Tweet aggregate**: a `TweetMedia` entity (`Url`/`PublicId`/`Position`) created **only** via `Tweet.AddMedia(...)` (internal ctor); `Tweet` exposes a read-only `Media` navigation backed by `_media` (the one place the Domain uses a navigation — it's an aggregate child, not a cross-aggregate reference, unlike author/parent). The FK (`TweetMedia.TweetId`, index, `OnDelete: Cascade`) is configured in `TweetConfiguration` (relationship, mapped via the backing field) + `TweetMediaConfiguration` (column lengths/index). **Single multipart endpoint**: `POST /api/tweets` is now `[Consumes("multipart/form-data")]` (`CreateTweetRequest`: `content`, optional `parentId`, optional `images` files) — the controller reads `IFormFile`s into `ImageUpload`s (so the Application never sees `IFormFile`), the handler uploads each then `tweet.AddMedia(...)`, and the tweet+media persist in one `SaveChanges`. Limits in `ImageUploadConstraints` (4 images, 5 MB each, jpeg/png/webp/gif) are enforced by `CreateTweetCommandValidator` **before** any upload → over-limit/wrong-type/oversized is a **400**; `[RequestSizeLimit(25 MB)]` caps the request. **A tweet must have content OR ≥1 image** (Twitter-style image-only tweets): the validator's `NotEmpty` on `Content` is conditioned `When` no image is attached, while the `≤280` cap always applies — so an image-only tweet persists an empty `Content` (allowed; the column's `IsRequired()` is NOT NULL, not non-empty). `TweetDto` gained `IReadOnlyList<TweetMediaDto> Media`, populated by `TweetRepository.Project` (ordered sub-select on the media navigation). Config: `Cloudinary:CloudName`/`ApiKey`/`UploadFolder` in `appsettings.json`, **`Cloudinary:ApiSecret` is a secret** (user-secret locally / `Cloudinary__ApiSecret` env on Render); all three credentials are **fail-fast validated** (`AddOptions<CloudinarySettings>().Validate(...).ValidateOnStart()`) so a misconfig aborts startup. Tests swap in a `FakeImageStorageService` (no network) with dummy Cloudinary config. **Note:** a partial upload that fails before `SaveChanges` can orphan Cloudinary assets (no cleanup yet). Migration `AddTweetMedia` (adds the `TweetMedia` table + FK/index — additive, no data loss).
- **Like + retweet engagement** (Module 3A, backend): an authenticated user can like/unlike and retweet/unretweet a tweet. `Like` and `Retweet` are **Domain** entities (in `Domain/Entities`) that reference the user and the tweet by plain `Guid` (`UserId`, `TweetId`) only — **no navigations**, so the Domain stays Identity-free (consistent with `RefreshToken`). Each has its own `I<Entity>Repository : IRepository<T>` (in `Application/Common/Interfaces`) exposing a single tracked `FindAsync(userId, tweetId)` for the write side, implemented in `Infrastructure/Persistence/Repositories` and registered `Scoped`. Configs (`LikeConfiguration`/`RetweetConfiguration`) set a **unique composite index on `(UserId, TweetId)`** (a user can like/retweet a tweet at most once — DB-level idempotency backstop), a non-unique index on `TweetId` (count lookups), and FKs to both `AspNetUsers` and `Tweets` with `OnDelete: Cascade` (deleting a user or a tweet removes its engagement rows). **Counts + "by me" flags are computed on the read side**, never denormalised: `TweetRepository.Project` gained correlated subqueries (`Context.Likes.Count(...)`, `Context.Retweets.Count(...)`) and `Any(... && UserId == currentUserId)` flags, so `TweetDto` carries `LikeCount`/`RetweetCount`/`LikedByCurrentUser`/`RetweetedByCurrentUser`. The flags need the caller, so **every read method takes a `Guid? currentUserId`** (`GetFeedAsync`/`GetRepliesAsync`/`GetByIdWithAuthorAsync`) sourced from `ICurrentUserService.UserId` in the query handlers — **`GET /api/tweets`, `/{id}`, and `/{id}/replies` stay public**, and an anonymous reader (`currentUserId == null`) gets both flags `false`. Endpoints (all `[Authorize]`, the user comes from the token): `POST`/`DELETE /api/tweets/{id}/like` and `POST`/`DELETE /api/tweets/{id}/retweet`, as CQRS commands `LikeTweet`/`UnlikeTweet`/`RetweetTweet`/`UnretweetTweet` (no validators — there's no body). Handlers are **idempotent**: like/retweet checks `ExistsAsync` (missing tweet → `NotFoundException` → **404**) then only stages a new row when `FindAsync` returns null; unlike/unretweet only stages a removal when one exists — so a double-like or double-unlike is a no-op success. Each command **returns the updated `TweetDto`** (re-read with the caller's id → **200** with the new count and flag) for a crisp client/Postman round-trip. **Idempotent under concurrency (Module 3 hardening):** the app-level `Find`-then-`Add`/`Remove` is backed by the DB. A concurrent double-like/retweet that loses the unique-index race, or a concurrent double-unlike/unretweet that finds the row already gone, must **not** 500 — `UnitOfWork.SaveChangesAsync` (the single commit point) translates the EF/Npgsql race exceptions into provider-agnostic Application exceptions (`UniqueConstraintViolationException` from a `23505` unique violation, `ConcurrencyConflictException` from a `DbUpdateConcurrencyException`/0-rows-affected), and the add/remove handlers **catch their respective one and treat it as the idempotent success it is** (the unique-violation detection is shared via `Infrastructure/Persistence/DbExceptions.IsUniqueViolation`, also used by `IdentityService`). The Application layer stays EF-free — it only ever sees the translated abstractions. Migration `AddLikesAndRetweets` (adds the `Likes` + `Retweets` tables, their indexes and FKs — additive, no data loss). **Deferred to later sub-steps:** follow + the Following feed (3B, where retweets surface in a feed), then the frontend (3C/3D).
- **Follow + Following feed** (Module 3B, backend): an authenticated user can follow/unfollow another **by handle**, read a lite user profile, and read a personal **Following feed**. `Follow` is a **Domain** entity (in `Domain/Entities`) — a directed edge `FollowerId → FolloweeId` referencing users by plain `Guid` only, **no navigations** (Identity-free, like `Like`/`Retweet`). `IFollowRepository : IRepository<Follow>` exposes a tracked `FindAsync(followerId, followeeId)` (write side); `FollowConfiguration` sets a **unique composite index `(FollowerId, FolloweeId)`** (one edge per pair — idempotency backstop; also serves the "does A follow B?" and following-list lookups), a `FolloweeId` index (follower-count side), and **two** cascade FKs to `AspNetUsers` (Postgres permits both cascade paths). **User reads** go through a new `IUserRepository` (NOT an `IRepository<T>` — the Identity user is not a Domain entity): `GetByHandleAsync(handle, currentUserId)` projects `AspNetUsers` + correlated follow counts into a `UserDto` (`Id`/`Handle`/`DisplayName`/`Bio`/`CreatedAtUtc`/`FollowerCount`/`FollowingCount`/`IsFollowedByCurrentUser`) — the Identity join lives in Infrastructure (`UserRepository`), mirroring `TweetRepository`, so Identity never crosses into Application; `GetIdByHandleAsync(handle)` resolves a handle for the write side. Handle matching is **case-insensitive and leading-@-tolerant** (see the Module 4B login/handles entry — both repo lookups compare on the normalized handle). Endpoints: `GET /api/users/{handle}` (**public**, flag reflects caller, 404 if unknown), `POST`/`DELETE /api/users/{handle}/follow` (**`[Authorize]`**, return the target's updated `UserDto`). CQRS: `FollowUser`/`UnfollowUser` commands + `GetUserByHandle` query (under `Application/Users/...`). Follow handler resolves the followee id (missing → `NotFoundException` → **404**), rejects **self-follow** by comparing ids and throwing a FluentValidation `ValidationException` (→ **400**), then idempotently stages the edge; unfollow is the mirror (idempotent, 404 on unknown handle). The **Following feed** (`GET /api/feed/following`, **`[Authorize]`** → **401** without a token; `FeedController` + `GetFollowingFeedQuery` under `Application/Feed/...`) is `TweetRepository.GetFollowingFeedAsync(currentUserId, cursor, limit)`: it **merges two timeline sources** — top-level tweets *authored* by followees and tweets *retweeted* by followees — each carrying its effective timeline time, via a `Concat` of lightweight `{ TweetId, SortTime }` rows. It then **de-duplicates to one row per tweet** at its **most-recent surfacing** (`GroupBy(TweetId).Select(Max(SortTime))`), so a popular tweet retweeted by several followees (and/or also authored by one) appears **once**, not many times. Keyset-paginated on `(SortTime, TweetId)` — after the GROUP BY, `TweetId` is unique per row, so it's the stable tiebreaker (the cursor encodes it). It loads the full `TweetDto` projections for just the page, plus the page's followee-retweets in one query, and **stamps `RetweetedBy`** from the *latest* surfacing: a retweet wins iff a followee retweeted the tweet at exactly the row's effective time (ties between an authored original and a retweet go to the retweet); otherwise the authored original shows and `RetweetedBy` stays null. **Deliberately deferred:** the aggregated **"X and N others retweeted"** label — 3B collapses to a single entry attributed to the most-recent retweeter only. `TweetDto` gained a nullable **`RetweetedBy`** (`RetweetedByDto`: `UserId`/`Handle`/`DisplayName`) — non-null **only** for a retweet entry in the Following feed; `TweetRepository.Project` passes `null` for it on every other read (main feed, replies, detail, create/like/retweet round-trips). Migration `AddFollows` (adds the `Follows` table + indexes/FKs — additive, no data loss). **Deferred:** the frontend wires like/retweet buttons (3C) and follow + the Following tab (3D).
- **"Who to follow" suggestions** (Module 3B follow-up, backend): `GET /api/users/suggestions?limit=` (**`[Authorize]`** → **401** without a token) returns a short list of `UserSuggestionDto` (`Id`/`Handle`/`DisplayName`/`AvatarUrl`/`Bio`/`FollowerCount`) for the "Who to follow" sidebar. CQRS `GetUserSuggestionsQuery` (under `Application/Users/Queries/...`; `DefaultLimit = 5`, `MaxLimit = 10`, clamped in the handler) → `IUserRepository.GetSuggestionsAsync(currentUserId, limit)`, implemented in `UserRepository`: a **single query** selecting users who are neither the caller nor already followed by them (`!Context.Follows.Any(f => f.FollowerId == currentUserId && f.FolloweeId == u.Id)`), follower count as a correlated subquery, **ordered by follower count descending** with `Id` as a stable tiebreaker — **no N+1**. **Gotcha (must order before projecting):** the `OrderByDescending` runs on the count subquery over the **entity** (`OrderByDescending(u => Context.Follows.Count(...))`), then `Take`, *then* `Select` into the DTO — ordering over a projected DTO member that wraps a subquery does **not** translate on PostgreSQL (runtime "could not be translated"). **No schema change / no migration** for suggestions itself; `AvatarUrl` was a placeholder `null` at 3B and is now populated from the stored avatar added in **Module 4A** (the client falls back to a placeholder when it is still null). Selection is deliberately simple (no recommendation algorithm). **Deferred / tracked (Module 3, not yet done):** the caller's own tweets in their own Following feed (you only see people you follow); public reads with an expired access token surfacing the caller's by-me flags; rate limiting on engagement/follow endpoints; and the "X and N others retweeted" aggregation. **Testing note:** the EF Core **in-memory** provider is non-relational and silently evaluates such queries client-side, so it does **not** catch translation failures; the repository query is covered by a **SQLite (in-memory, relational)** test (`UserSuggestionsSqliteTests`) that exercises real SQL translation. The test project pins `SQLitePCLRaw.bundle_e_sqlite3` 3.x to avoid the advisory (NU1903) in the version EF's SQLite provider pulls transitively.
- **Profile: timelines, edit & avatar** (Module 4A, backend): a user's profile (by handle) now exposes their own tweets and their liked tweets, the logged-in user can edit their `displayName`/`bio` and upload a real **avatar** (Cloudinary), and the avatar URL flows into every read that carries it. **Avatar storage:** `ApplicationUser` gains a nullable `AvatarUrl` (`MaxAvatarUrlLength = 500`, configured in `ApplicationUserConfiguration`); migration `AddUserAvatar` (additive column, no data loss). A companion nullable `AvatarPublicId` (`MaxAvatarPublicIdLength = 255`; migration `AddAvatarPublicId`, additive) stores the image host's public id alongside the URL **so a replaced/removed avatar's old asset can be deleted** (no orphaned Cloudinary images); it is never surfaced in any DTO. `AvatarUrl` is surfaced on `UserDto` (profile) and populated on `UserSuggestionDto`; it also rides on `AuthUser` → `CurrentUserDto` so `GET /api/auth/me` carries the caller's avatar. **Timelines** (both **public**, cursor-paginated `CursorPage<TweetDto>`, **404** when the handle is unknown): `GET /api/users/{handle}/tweets` = the user's **top-level authored** tweets newest-first (`ITweetRepository.GetUserTweetsAsync`, same `(CreatedAtUtc, Id)` keyset as the main feed, filtered by `AuthorId` and `ParentId == null`); `GET /api/users/{handle}/likes` = the tweets they **liked**, **most-recently-liked first** (`GetUserLikedTweetsAsync` — keyset over the **`Like` row's** `(CreatedAtUtc, TweetId)`, since a like has its own time the tweet projection can't carry; it pages the like rows, loads the `TweetDto` projections for just that page in one query, then re-stitches in like-order and builds the `nextCursor` from the last like — mirroring `GetFollowingFeedAsync`). Both queries resolve the handle via `IUserRepository.GetIdByHandleAsync` (unknown → `NotFoundException` → 404) and pass `ICurrentUserService.UserId` for the by-me flags (anonymous reader ⇒ flags false). - **Notifications: real-time push via SignalR** (Module 5B, backend): when a notification is created, the recipient's connected clients receive the new `NotificationDto` + their updated unread count in real time — no polling. SignalR is an **API** concern behind an **Application** abstraction `INotificationPublisher` (`PublishAsync(recipientId, NotificationPushDto)`; `NotificationPushDto { notification, unreadCount }` in `Application/Notifications`) — neither Application nor Infrastructure references SignalR. **Push is wired at the single commit chokepoint, not in the four handlers:** `UnitOfWork.SaveChangesAsync` snapshots the `Notification` entries `Added` in the transaction, commits, then **after a successful commit** best-effort pushes each (re-projects via `GetProjectedAsync`, reads `GetUnreadCountAsync`, calls the publisher) — a delivery failure is swallowed (logged) and never rolls back the action. Self-skipped / de-duplicated actions stage no row, so they are naturally never pushed. The hub `NotificationHub : Hub<INotificationClient>` (push-only; clients call no server methods) is mapped at **`/hubs/notifications`** with `[Authorize]`. **JWT over WebSocket:** browsers can't set the `Authorization` header on a WebSocket, so the token is read from the **`access_token` query string** — accepted **only for the hub path** (`OnMessageReceived` in `AddJwtBearerAuthentication`; REST still requires the header, to avoid query-string token leakage). A connection is keyed to its user by the JWT **`sub`** claim via a custom `SubClaimUserIdProvider` (the default reads `ClaimTypes.NameIdentifier`, but `MapInboundClaims = false` keeps the claim named `sub`), so `Clients.User(recipientId.ToString())` targets exactly that person across all their devices. CORS already allows credentials with an explicit origin list (required for the cross-site socket). No schema change / no migration. **Testing:** `NotificationsRealtimeTests` swaps in a recording `INotificationPublisher` (no live socket) to assert the chokepoint pushes the right payload/recipient/unread-count on a like, pushes **nothing** on a self-action or a de-duplicated repeat-like, and that the hub's negotiate handshake is **401 without a token** but **200 with the token in the query string**. **Deferred:** the bell/badge/list **UI** and the client connection (5C); the retweeter avatar / "X and N others" aggregation remain deferred from Module 3/4.

- **Mentions: parse + notify** (Module 9A, backend): posting a tweet/reply whose text contains `@handle` creates a **`Mention`** notification for each mentioned user, **reusing the Module 5 notification system** (so 5B's commit-chokepoint push delivers it in real time, and 5A's list/unread-count/mark-read all cover it for free). **No migration:** the `Notification.Type` enum is stored as an `int`, so adding `Mention = 4` to `NotificationType` (kept `[JsonStringEnumConverter]` ⇒ serializes as `"Mention"`) is purely additive — no schema change. **Hashtags need no backend in v1** (9B makes `#tag` clickable → the existing tweet search), so this sub-step is mentions only. **Parsing** is pure text logic in the Application layer: `Application/Common/MentionParser.ExtractHandles(content)` (a `[GeneratedRegex]` `(?<![A-Za-z0-9_])@([A-Za-z0-9_]+)` — the leading negative lookbehind means an email's `@` is **not** a mention) returns the distinct handles (case-insensitive, first-appearance order; no Identity/DB dependency, so it stays in Application). **Resolution** is a new batch lookup `IUserRepository.GetIdsByHandlesAsync(handles)` → distinct user ids, implemented in `UserRepository` by normalising each handle via `HandleNormalizer` and matching `NormalizedHandle` with a single `Contains` ⇒ `IN (...)` query (translates on both Npgsql and SQLite); unknown handles are simply absent. **Wiring** lives in `CreateTweetCommandHandler` (now also injects `IUserRepository`): after staging the tweet, it parses the content, resolves the ids, and calls the existing `INotificationService.CreateAsync(recipient, actor, NotificationType.Mention, tweet.Id)` for each — staged in the **same `SaveChanges`** as the tweet, with `TweetId` pointing at the tweet/reply itself (so the notification previews it). - **Quote tweet: backend** (Module 10A, backend): a **quote tweet** is a normal **top-level** tweet that **embeds another tweet** (content + a reference to the quoted one) — distinct from a retweet (pure reshare, no text) and a reply (`ParentId`). **Schema:** `Tweet` gains a second nullable self-reference **`QuotedTweetId`** (independent of `ParentId`; ctor is now `(content, authorId, parentId = null, quotedTweetId = null)`) — still **navigation-free** (the Domain carries only the id). `TweetConfiguration` adds an index on `QuotedTweetId` and a self-FK with **`ON DELETE SET NULL`** (the key difference from the `ParentId` cascade): deleting the quoted tweet **nulls** the reference in the quotes of it so the quote **survives and renders "unavailable"**, rather than cascade-deleting the quotes or FK-erroring on our hard delete. Migration `AddQuoteTweet` (additive: column + index + SET-NULL FK). **Create:** `POST /api/tweets` accepts an optional **`quotedTweetId`** form field (alongside `content`/`parentId`/`images`); `CreateTweetCommand` gains `QuotedTweetId`. A quote **must have text** (it's a comment on the quoted tweet) — enforced by the validator (`Content NotEmpty When QuotedTweetId.HasValue`) → **400**. A **missing quoted id is a 404** (not 400): the existence check lives in `CreateTweetCommandHandler` (validators only yield 400), done **up front** via `GetAuthorIdAsync` (one query that also serves the notification) **before any image upload** so a bad id fails fast without orphaning uploads. Quotes are **top-level** (don't combine quote+reply into a compound type); **self-quote is allowed** (no self-notification — the service self-skips). **Preview (one level, no recursion):** `TweetDto` gains **`QuotedTweet`** (`QuotedTweetDto`: `Id`/`Content`/`Author { Handle, DisplayName, AvatarUrl }`/`CreatedAtUtc`/`Media`) — a lightweight embedded card, **non-recursive by construction** (`QuotedTweetDto` has **no** nested `QuotedTweet`/`Parent`/counts, so a quote-of-a-quote previews exactly one level). It's populated in the **shared `TweetRepository.Project`** by a single correlated subquery (`from q in Context.Tweets where q.Id == tweet.QuotedTweetId join author … select … FirstOrDefault()`, with an ordered nested `Media` sub-select) — **one join, no N+1**, so it rides **every** read path (feed, replies, detail, following feed, profiles, search). It is **`null`** when the tweet isn't a quote **or** the quoted tweet was deleted (SET NULL nulls the id, and even a dangling id simply misses the join ⇒ `FirstOrDefault()` returns null ⇒ the client shows "This post is unavailable"). `TweetDto` also gains **`QuoteCount`** (tweets quoting this one — a correlated count, like the other counts). **Notification:** `NotificationType` gains **`Quote = 5`** (additive enum value, string-serialized; **no migration** — the column is an int); on creating a quote, `CreateTweetCommandHandler` calls the existing `INotificationService.CreateAsync(recipient = quoted author, actor = quoter, Quote, tweetId = the quote tweet)` — same `SaveChanges`, self-skip + unread-dedup reused, **real-time via 5B** (the commit-chokepoint push), click-through target is the quote tweet. (A quote that also `@`-mentions the quoted author yields both a Quote and a Mention — quote-vs-mention de-dup was **not** in scope, unlike the reply-vs-mention skip.) **Testing:** `QuoteTweetSqliteTests` (relational — the one-level preview projection + nested `Media` collection translate, `QuoteCount` correct, non-quote ⇒ null preview, quote-of-a-quote previews one level, deleting the quoted tweet ⇒ null preview while the quote survives; sanity-checked to catch a translation break) + `QuoteTweetApiTests` (HTTP spine — quote embeds the preview on detail+feed, notifies the quoted author with `tweetId` = the quote, self-quote ⇒ no notification, delete quoted ⇒ null preview, quote needs content ⇒ 400, missing quoted id ⇒ 404). **Deferred:** 10B builds the quote UI (Quote option on the retweet button, compose with an embedded preview, the embedded quoted card in the feed).
- **Tweet edit: backend** (Module 11A, backend): the author can edit their own tweet's **text** within a configurable time window (Twitter-style), after which it carries an `editedAtUtc` so the UI can show an "edited" marker. **No edit history**, **no media edit** (text only), and **no mention re-notify** — all deferred. **Schema:** `Tweet` gains a nullable **`EditedAtUtc`** (null until first edited) and a domain method `Edit(content, editedAtUtc)` that updates `Content` + stamps `EditedAtUtc` only (author/parent/quoted/media/counts and **`CreatedAtUtc` are never touched**). Migration `AddTweetEdited` (additive column). `TweetDto` gains **`EditedAtUtc`**, populated from the same row in the shared `TweetRepository.Project` → it rides every read path. **Endpoint:** `PUT /api/tweets/{id}` (**`[Authorize]`**, JSON `{ content }`) → `EditTweetCommand(Id, Content)` → updated `TweetDto`. **Rules & status codes** (checked in `EditTweetCommandHandler`, mirroring delete's author-auth): not found → **404**; **author-only** (`tweet.AuthorId != ICurrentUserService.UserId`) → `ForbiddenAccessException` → **403**; **edit window** — only within `EditWindow` of `CreatedAtUtc`, else `EditWindowExpiredException` → **409 Conflict** (deliberately **409, not 403**: the caller IS the rightful author, so it's the tweet's *age* that conflicts with the edit, distinct from the non-author 403; new `EditWindowExpiredExceptionHandler` registered in `Program.cs`). Content is validated by `EditTweetCommandValidator` (non-empty + ≤`MaxContentLength`) → **400** (edit is a text op, so content is always required — unlike create's media-or-text rule). The handler loads the tweet untracked (`GetByIdAsync`), mutates, and `Update()`s it (same untracked-then-stage style as `DeleteTweetCommandHandler`), then re-reads via the author join for the response. **No mention re-notify:** editing does **not** re-parse/re-notify mentions — mentions added via an edit are intentionally not notified in v1. **Config:** the window is **`TweetEditSettings`** (`Application/Common/Models`, `WindowMinutes` default **30**, exposes `EditWindow`), bound from the `TweetEdit` config section into a **plain-POCO singleton** in `AddInfrastructure` (so the Application handler depends on the settings type directly — no `IOptions`/configuration package leaks into Application; missing section ⇒ the 30-min default). **Testing:** `EditTweetApiTests` (HTTP spine — author edits within the window sets `editedAtUtc` & new content, other readers see it, `CreatedAtUtc` unchanged; non-author → 403; **past-window → 409** by backdating `CreatedAtUtc` in the shared in-memory store; empty → 400; missing → 404; an edit that adds a mention does **not** notify and leaves author/createdAt/likeCount unchanged). `editedAtUtc` is a trivial scalar on the existing projection (no new query shape), so the existing `TweetReadsSqliteTests` cover its translation. **Deferred:** 11B adds the edit UI (an "Edit" item in the tweet ⋯ menu within the window, the edit composer, and the "edited" marker on the card).
- **Direct messages: data + REST foundation** (Module 12A, backend): the data + REST spine for **1-on-1, text-only** DMs (anyone can DM anyone). **No real-time yet** (12B layers SignalR on top, reusing the 5B hub pattern); the chat UI is 12C. **Entities** (Domain, Identity-free, Guid refs, no navigations — like `Like`/`Follow`): **`Conversation`** (`Id`/`CreatedAtUtc`, `LastMessageAtUtc` sort key, and a canonical **`PairKey`** `"{minUserId}:{maxUserId}"` built by `Conversation.BuildPairKey` with a **unique index** → exactly one conversation per pair); **`ConversationParticipant`** (composite PK `(ConversationId, UserId)`, nullable **`LastReadAtUtc`** = this member's read state, `MarkReadAt(now)`; **not** a `BaseEntity` — so it's staged via `Context.Set<>` in the repo, not the generic `IRepository`); **`Message`** (`BaseEntity` + `ConversationId`/`SenderId`/`Content`, `MaxContentLength = 2000`). FKs all cascade to `AspNetUsers`/`Conversation` in Infrastructure; indexes `Conversation.PairKey` (unique), `Conversation.LastMessageAtUtc`, `ConversationParticipant.UserId` ("my conversations"), `Message.(ConversationId, CreatedAtUtc)` (thread). Migration `AddDirectMessages` (additive). **Repositories:** `IConversationRepository : IRepository<Conversation>` (get-or-create helpers `GetIdByPairKeyAsync`/`AddParticipantAsync`, authz `IsParticipantAsync`/`ExistsAsync`, tracked getters for the recency bump + mark-read, and the read projections) + `IMessageRepository : IRepository<Message>` (the thread read + single-message DTO for the send response). The list/message projections do the Identity join in Infrastructure (so Application sees only DTOs) — `ChatUserDto`/`ConversationDto`/`MessageDto`/`LastMessagePreviewDto`/`UnreadConversationsDto` under `Application/Conversations`. **Endpoints** (all on `ConversationsController`, **class-level `[Authorize]`**, actor always from `ICurrentUserService`): `POST /api/conversations` (`{ recipientHandle }` or `{ recipientUserId }`) **get-or-creates** the pair's conversation — **idempotent**, mirroring the engagement race handling: pre-check `GetIdByPairKeyAsync`, else create conversation + both participants in one `SaveChanges`, **catch `UniqueConstraintViolationException` → re-read by `PairKey`** (the unique index makes concurrent creates converge to one); **can't DM yourself → 400** (FluentValidation `ValidationException`), **unknown recipient → 404**; `POST /api/conversations/{id}/messages` (`{ content }`) sends text (participant-only, bumps `LastMessageAtUtc`) → **201**; `GET /api/conversations` lists the caller's conversations **ordered by `LastMessageAtUtc` desc**, keyset over `(LastMessageAtUtc, Id)` (reusing `TweetCursor`) via the **two-step page-then-project** pattern (order the conversation rows, then project just the page — no N+1, ordering never over a subquery-wrapped member), each `ConversationDto` carrying **otherParticipant**, **lastMessage** preview (nullable), and **unreadCount**; `GET /api/conversations/{id}/messages` pages messages **newest-first** keyset over `(CreatedAtUtc, Id)` (participant-only); `POST /api/conversations/{id}/read` sets the caller's `LastReadAtUtc = now` → `{ unreadCount: 0 }`; `GET /api/conversations/unread-count` → the **DM badge**. **Unread definitions:** a conversation's `unreadCount` = messages with `CreatedAtUtc > my LastReadAtUtc` (or all, when `LastReadAtUtc IS NULL`) **and `SenderId != me`** (my own messages never count); the **badge** is the count of **conversations that have ≥1 unread message** (conversations-with-unread, the simplest badge-friendly choice — not a raw message total). **Participant-only authorization everywhere:** a non-participant gets **403** on read/send/mark-read (a missing conversation → **404**, so existence isn't leaked beyond the 403 vs 404 split), and a conversation never appears in a non-member's list. **DMs do NOT use the notification system** (no `Notification` rows for messages — they have their own unread/badge). **Testing:** `ConversationsSqliteTests` (relational — the list projection's other-participant join + last-message sub-select + the per-caller unread correlated subquery incl. `LastReadAtUtc IS NULL`, recency keyset, and the newest-first messages query with `isMine`/no-dupes; sanity-checked to catch a translation break) + `DirectMessagesApiTests` (HTTP spine — the full A↔B flow: start → send → B's list shows unread 1 + preview → B reads/marks → unread 0 → B replies → A sees it; get-or-create idempotent whoever starts; self-DM 400; unknown 404; empty content 400; recency bump; **participant isolation** — C is 403 on read/send/mark-read and the conversation isn't in C's list; 401s). **Deferred:** 12B SignalR `ChatHub` (real-time delivery, reusing the 5B JWT-over-socket + push-after-commit chokepoint) and 12C the conversation-list + chat-thread UI; group conversations, image/attachment messages, typing indicators, per-message read receipts, and a "followers-only DM" restriction are out of scope.
- **Direct messages: real-time push** (Module 12B, backend): when a message is sent, the other participant receives it live and their DM badge updates — **reusing the 5B notification real-time pattern wholesale**, no second mechanism. **`ChatHub : Hub<IChatClient>`** at **`/hubs/chat`** (`[Authorize]`, push-only; `IChatClient.ReceiveMessage(MessagePushDto)`), mapped in `Program.cs` next to `NotificationHub`. **Same socket auth:** the JWT arrives in the **`access_token` query string**, accepted by the *same* `OnMessageReceived` predicate in `AddJwtBearerAuthentication` — extended from `path.StartsWithSegments(NotificationHub.Path)` to also match `ChatHub.Path` (one rule, both hubs; REST still requires the `Authorization` header). Connections are keyed to the user by the JWT **`sub`** claim via the existing `SubClaimUserIdProvider`, so `Clients.User(recipientId)` targets that person across devices. **Publisher abstraction** `IMessagePublisher` (Application, framework-free, mirrors `INotificationPublisher`) with its SignalR impl **`ChatHubPublisher`** in the **API** layer (`IHubContext<ChatHub, IChatClient>`) — Infrastructure/Application stay SignalR-free. **Push-after-commit at the UoW chokepoint:** `UnitOfWork.SaveChangesAsync` now also snapshots the `Message` rows `Added` in the transaction (alongside notifications), and **after a successful commit** fans each out — resolves the conversation's participants via `IConversationRepository.GetParticipantUserIdsAsync`, **skips the sender**, projects the message **for each recipient** (`IMessageRepository.GetDtoAsync(messageId, recipientId)` ⇒ `isMine = false`), reads their `GetUnreadConversationCountAsync`, and calls `IMessagePublisher.PublishAsync(recipientId, new MessagePushDto(dto, conversationId, unreadCount))`. **Best-effort** (try/catch, logged) — a push failure never rolls back or fails the send (already committed; REST already returned 201); never pushed before/inside the commit. **Sender echo is deferred** (the sender's UI send is optimistic; multi-device echo is a later nice-to-have) — only the *other* participant(s) are pushed, exactly as notifications never push the actor. **Client contract (for the frontend wiring):** connect to `{API_ORIGIN}/hubs/chat?access_token={JWT}`; subscribe to event **`ReceiveMessage`** with payload `{ message: MessageDto, conversationId, unreadCount }` (`message` is the 12A `MessageDto` — `id`/`conversationId`/`senderId`/`sender{handle,displayName,avatarUrl}`/`content`/`createdAtUtc`/`isMine`). No schema change / no migration. **Testing:** `DirectMessagesRealtimeTests` swaps in a recording `IMessagePublisher` (extending the 5B `RecordingPublisherWebAppFactory`) to assert a send pushes the right `ReceiveMessage` payload + unread count to the **recipient only** (the sender isn't echoed), and that the hub's negotiate is **401 without a token** but **200 with the token in the query string**; `DirectMessagesPublishResilienceTests` uses a throwing publisher to confirm a failed push **doesn't break the send** (201 + the message is still readable). **Deferred:** 12C wires the frontend SignalR client into the `ConversationsProvider` seam (`pushIncomingMessage`); typing indicators / read-receipt broadcasts remain out of scope.

## Adding a feature (the pattern)

1. (If needed) add/extend a Domain entity + its `IEntityTypeConfiguration`. For a new aggregate, add an `I<Entity>Repository : IRepository<Entity>` interface in `Application/Common/Interfaces`, implement it in `Infrastructure/Persistence/Repositories` (inherit `Repository<Entity>`), and register it `Scoped` in `Infrastructure/DependencyInjection`.
2. Add a command/query record implementing `IRequest<TResult>` under `Application/<Feature>/...`.
3. Add the handler (inject the repository + `IUnitOfWork`; commands stage changes then call `SaveChangesAsync`) and, for commands, a FluentValidation validator beside it.
4. Add a DTO record for the result.
5. Add or extend a thin controller action that sends the request.
6. If the schema changed, add an EF migration (see below).

## Testing policy (relational coverage for non-trivial queries)

The HTTP integration tests run on the EF Core **in-memory** provider, which is **non-relational**: it evaluates LINQ in process and so **silently passes queries that cannot be translated to SQL** (e.g. ordering/filtering over a projected DTO member that wraps a subquery — this exact bug shipped once in `GetSuggestionsAsync` and only failed at runtime on PostgreSQL). The in-memory provider is fine for proving the HTTP spine, but it is **not** a safety net for SQL translation.

**Policy — applies to every module from here on:** any repository query that involves **subqueries** (correlated counts, `Any`/`Count`), **projections to a DTO**, **collection projections** (e.g. `Media`), **unions/`Concat`**, **keyset pagination** (`Guid.CompareTo`/cursor predicates), or anything non-trivial **MUST** have a **relational test** — an in-memory **SQLite** test at the repository level (SQLite is relational, so the query is really translated to SQL). Use the `SqliteTestHarness` (opens a shared in-memory SQLite connection, `EnsureCreated`, seed via one `ApplicationDbContext`, query via another). Assert both that it **translates** (runs without throwing) and that it returns **correct results** (contents, ordering, pagination with no duplicates/skips). Existing examples: `FollowingFeedSqliteTests`, `TweetReadsSqliteTests` (feed/replies/detail), `UserProfileSqliteTests`, `UserSuggestionsSqliteTests`. When adding such a test, sanity-check it actually catches a translation break by temporarily introducing a non-translatable construct (e.g. `Guid.ToByteArray()` in a predicate) and confirming the test fails with "could not be translated", then revert. The SQLite provider's native bundle is pinned (`SQLitePCLRaw.bundle_e_sqlite3` 3.x) in the test project to avoid the advisory in the version EF pulls transitively.

## Database & config

- PostgreSQL via Npgsql. Local connection comes from `ConnectionStrings:DefaultConnection` in `appsettings.json`; `DATABASE_URL` (a `postgres://…` URI or plain connection string) overrides it and is what Render supplies.
- The app runs `Database.Migrate()` on startup (guarded by `Database.IsRelational()`, so the in-memory test provider skips it), so deploys self-provision the schema.
- Both the runtime DI registration and the design-time `ApplicationDbContextFactory` resolve the connection string through the single `ConnectionStringResolver` (URI/SSL handling lives in one place — they can't drift). The factory lets `dotnet ef` work without booting the API host.
